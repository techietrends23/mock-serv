1. Setup & Installation

Prerequisites

  • Node.js 20+ and npm 10+
  • macOS, Windows, or Linux

Install

git clone <repo-url> mock-serv
cd mock-serv
npm install
npx playwright install chromium
Note: Playwright is required for the Traffic Capture feature. The app works without it for mock import/editing.

Verify

npm run check
npm test

2. Running the App

Development mode

1

Start the app

Run both the API server and web UI together:

npm run dev
2

Open the UI

Check the terminal output for the Vite URL (typically http://127.0.0.1:5173 or the next available port).

Production mode

npm run build
npm run start

This serves the built UI from http://127.0.0.1:3001 — everything in one process.

What runs where

Component Port Description
Web UI (Vite)auto (first available)React frontend with the mock editor
API Server (Fastify)3001 (configurable via MOCK_SERV_PORT)Local backend — persistence, mocks, capture
Mock ServersdynamicEach running mock gets its own port

3. Traffic Capture — Create Mocks from Live Traffic

Mock Serv can open a browser, record every HTTP request it makes, and turn any individual request into a reusable mock — no OpenAPI specs or config files needed.

How it works

  1. A headed Chromium browser window opens on your machine
  2. You navigate to the app or website you want to mock
  3. Playwright captures every request and response pair
  4. Captured calls are stored in SQLite and shown in the UI
  5. Select any call and click Create Mock to instantly generate a mock

Step-by-step

1

Open Traffic Capture

In the sidebar, click the Traffic Capture button to expand the capture panel.

2

Create a session

Type a name (e.g. my-app) and click + New Session.

3

Start capturing

Click Start on the session. A Chromium browser window opens automatically.

4

Navigate to your target

In the browser that opened, type the URL of the app you want to mock and press Enter. Every network request the page makes is captured.

5

Stop the session

Return to Mock Serv and click Stop. The captured calls appear in a list.

6

Select a call

Click any call to inspect its full request and response:

  • Request method, URL, and headers
  • Request body (JSON formatted)
  • Response status code and headers
  • Response body
7

Create a mock

Click Create Mock. In the dialog, optionally name the mock and confirm. Mock Serv automatically:

  • Extracts the HTTP method and path
  • Infers query parameters from the URL
  • Infers request & response JSON schemas from bodies
  • Sets the response status code and example body
  • Creates a full MockDefinition with one endpoint
8

Use the mock

The new mock appears in Available Mocks. Click Start to boot it, then call it:

curl http://127.0.0.1:<port>/your/path
Tip: You can create mocks from multiple calls in the same session. Each mock gets its own port and can be edited independently.

4. Import API Definitions

Mock Serv can parse existing API definitions and convert them into editable mocks. Supported formats:

Format Source type Example
OpenAPI / Swagger openapi YAML or JSON spec files
cURL curl curl -X POST https://api.example.com/users -H ...
Postman Collection postman Exported Collection v2.1 JSON
HAR har Browser-exported HTTP Archive

Steps

1

Select source type

Choose from OpenAPI, cURL, Postman, or HAR in the Import panel.

2

Choose protocol

Select REST or GraphQL depending on your API.

3

Provide the definition

Either paste the content directly into the textarea, or click Choose local file to upload.

4

Name and import

Enter a mock name (it must be unique) and click Analyze and Import.

5

Edit and start

The mock appears in Available Mocks. Select it to edit endpoints, then click Start to serve it.

5. Managing Mocks

Mock Editor

When you select a mock from the list, the main panel splits into two columns:

Editor (left) Inspector (right)
Mock name, protocol, port Runtime status and port
Global latency & error rate Endpoint count
Endpoint list (expand/collapse) Selected endpoint detail
Per-endpoint: method, path, status, schemas, headers Recent logs
Add / remove endpoints

Actions

  • Start — boots a local HTTP server on a random (or specified) port
  • Stop — gracefully shuts down the mock server
  • Hot Reload — restarts a running mock after edits
  • Delete — removes the mock and all its data from SQLite
  • Save — persists all editor changes
  • Seed Data — inserts sample rows into the mock's CRUD table for an endpoint

Endpoint editing

Each endpoint can be configured individually:

  • Method — GET, POST, PUT, PATCH, DELETE, OPTIONS, HEAD
  • Path — supports path parameters with :param or {param} syntax
  • Status code — the HTTP status to return
  • Latency — artificial delay in milliseconds
  • Error rate — probability (0–1) of returning an error instead of the normal response
  • Request headers — JSON object of header key/value pairs
  • Request schema — JSON Schema for validating incoming request bodies
  • Response schema — JSON Schema that defines the shape of generated responses
  • Response example — a concrete response body (used as-is when no schema is provided)

6. Petstore Example

The repo includes petstore.yaml — a full OpenAPI 3 Petstore specification. Use it to verify the end-to-end workflow:

Manual walkthrough

1

Start the app

npm run dev
2

Import petstore.yaml

In the Import panel: set Source OpenAPI / Swagger, Protocol REST, upload petstore.yaml, enter name petstore-local, click Analyze and Import.

3

Start the mock

Click Start on the petstore-local card. Note the assigned port (e.g. 54321).

4

Call the mock

# List pets by status
curl http://127.0.0.1:54321/pet/findByStatus?status=available

# Get a pet by ID
curl http://127.0.0.1:54321/pet/1

# Create a pet
curl -X POST http://127.0.0.1:54321/pet \
  -H 'content-type: application/json' \
  -d '{"id":1,"name":"Fluffy","status":"available"}'

# Update a pet
curl -X PUT http://127.0.0.1:54321/pet \
  -H 'content-type: application/json' \
  -d '{"id":1,"name":"Fluffy","status":"sold"}'

# Store inventory
curl http://127.0.0.1:54321/store/inventory

Automated verification

npm run verify:petstore

This script starts the API, imports petstore.yaml, boots the mock, hits GET /pet/findByStatus, and asserts a 200 response with an array body.

7. Mock Data & CRUD Rows

How mock responses work

When a mock endpoint receives a request, it generates a response using this priority:

  1. Response example — if set, returned as-is for every request
  2. Response schema — generates sample data matching the schema
  3. CRUD rows — for seeded endpoints, reads from the row table
  4. Fallback — returns {} or a basic placeholder

Seeding data

Seeded data gives you predictable responses for CRUD-style endpoints. To seed data:

1

Select a mock and expand an endpoint in the editor.

2

Click Seed Data — it inserts a sample row into the endpoint's CRUD table.

3

The next GET request returns the seeded rows. POST creates new rows, PUT/PATCH update them, DELETE removes them.

Example: CRUD workflow

After seeding a row for a GET /users endpoint:

Request Response
GET /users [{"id":"sample","name":"Seed Row","active":true}]
GET /users/sample {"id":"sample","name":"Seed Row","active":true}
POST /users with {"name":"New"} {"id":"row_abc123","name":"New"}
PUT /users/sample with {"name":"Updated"} {"id":"sample","name":"Updated","active":true}
DELETE /users/sample 204 No Content

Schema-driven responses

If you set a response schema instead of (or in addition to) an example, Mock Serv generates realistic data:

Example response schema

{
  "type": "object",
  "properties": {
    "id": { "type": "integer", "example": 42 },
    "email": { "type": "string", "format": "email", "example": "user@example.com" },
    "profile": {
      "type": "object",
      "properties": {
        "displayName": { "type": "string" },
        "avatar": { "type": "string", "format": "uri" }
      }
    }
  }
}

With this schema, GET /users/42 returns:

{
  "id": 42,
  "email": "user@example.com",
  "profile": {
    "displayName": "string",
    "avatar": "string"
  }
}
Tip: Use the Traffic Capture feature first to see real response bodies, then Create Mock to automatically generate the schema from those real responses.

8. Tips & Troubleshooting

Best practices

  • Use Traffic Capture for new APIs — the quickest way to create mocks is to record a real session
  • Name mocks consistently — names become part of SQLite table names, so keep them identifier-safe
  • Set latency for realistic behavior — add 50–200ms latency to simulate network conditions
  • Use error rates for testing — set errorRate: 0.1 to make 10% of responses fail
  • Hot reload after edits — click Hot Reload to apply endpoint changes to a running mock

Common issues

Issue Solution
UI loads but actions fail Ensure both frontend and backend are running: npm run dev. Check http://127.0.0.1:3001/api/health.
Traffic Capture browser doesn't open Run npx playwright install chromium to install the browser binary.
Capture shows no calls Make sure you navigated to a URL in the browser window that opened, and that the target sends HTTP requests.
Port conflict Change ports with MOCK_SERV_PORT=3002 or MOCK_SERV_HOST=0.0.0.0.
OpenAPI import fails The spec must be valid. External $ref targets must be resolvable over the network.
Mock starts but returns errors Check the Logs panel in the inspector for runtime error messages.
